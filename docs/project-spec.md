# Genetics Results API - Project Specification

## Purpose

A FastAPI-based REST API that serves human genetics association results and annotations. It provides read-only access to tabix-indexed data files stored on Google Cloud Storage, enabling queries by genomic region, variant, gene, and phenotype across multiple genetic studies and datasets.

## Architecture

### Tech Stack
- **Python 3.13** with **FastAPI** and **uvicorn**
- **htslib/tabix** for indexed genomic data access (compiled in Docker image)
- **Google Cloud Storage** for data files (accessed via gcloud-aio-storage, gcsfs)
- **Polars** for data manipulation
- **JWT** authentication (PyJWT)
- **uv** for dependency management
- Deployed as a Docker container (base: nikolaik/python-nodejs)

### Data Access Pattern

1. **Configuration-driven**: Each data resource is defined in `app/config/` with paths to GCS-hosted tabix files and column mappings. Dataset metadata (author, version, description, metadata_file, harmonizer type) is loaded from a shared `configs/datasets.yaml` file via `app/config/yaml_loader.py`, which selects the active profile from the `CONFIG_PROFILE` env var and reads the YAML path from `DATASETS_CONFIG_PATH` (default: `./configs/datasets.yaml`). The canonical YAML lives in the suite repo and is synced to each service repo for local dev. Product configs (credible_sets, exome_results, gene_based_results, summary_stats) reference the registry via `dataset_id` and contain only file paths and product-specific settings.
2. **Range queries (in-process tabix)**: Genomic region/variant range queries no longer shell out to the `tabix` subprocess. `GCloudTabixBase._stream_range` parses the bgzf `.tbi` index in Python (`app/services/tabix_query.py`), turns the query regions into the minimal set of compressed byte ranges, fetches them **concurrently over the pooled aiohttp GCS session** (a `Range:` header GET per range, bounded by `_RANGE_CONCURRENCY`), decompresses the bgzf blocks with stdlib `zlib`, and filters to records overlapping the regions — producing the same records `tabix -R` would, in coordinate order. This replaces the old single serial `tabix` subprocess that issued ~one HTTP range request per region (the dominant latency for large variant batches): an 888-variant single-resource query dropped ~18 s → ~4–5 s and small batches ~1.5 s → ~0.3 s. See "GCS access" below for details (header retrieval still uses `tabix -H`). Indexed `.tsv.gz` files remain on GCS. Exome variant results are served for genebass (filtered to mlog10p > 4), IBD exome (resource: `ibd_exome_2026`, 3 phenotypes: IBD, UC, CD, filtered to mlog10p > 4). Data files are in `exome_results/` on GCS. Individual trait exome variant results can be queried by phenotype via the `exome_results_by_phenotype` endpoint, which uses prefix/suffix patterns to locate per-trait data files. Exome variant columns include `n_cases`/`n_controls` (nullable) and `trait_original`. Gene-based burden results use a separate column schema with `trait_original` and `flags` columns. When several datasets share one combined file (e.g. the external pseudo CS file bundling `covid_hgi`, `pgc_scz`, `pgc_bip`, `gp2_pd` under `credible_sets/ext/`), `DataAccess.stream_range*` dedups by `all_cs_file`/`all_exome_file` so tabix runs once, and `tsv_line_iterator` is given a `resource_filter` that drops shared-file rows belonging to resources not in the request. Per-phenotype individual files live in per-dataset subdirs `ext/individual/<dataset_id>/` so that datasets sharing a resource (e.g. `pgc_scz` + `pgc_bip` → `pgc`) don't collide on phenotype lookups. Summary-stats datasets are served from per-phenotype `prefix`+`suffix` files keyed by phenotype; `finngen_ukbb_pqtl` (resource `finngen_ukbb`, `data_type=pqtl`, v1) is a 3-way inverse-variance pQTL meta-analysis of FG2-3K (Olink 3K) + UKB-PPP (Olink 3K) + FG3 Cross-Batch (Olink HT 5K) over 2,831 proteins (max combined N=48,703, GRCh38/dbSNP b156), keyed by gene symbol with per-study and combined-meta columns. It is summary-statistics only (no credible sets/colocalization — `products` auto-derives `summary_stats` for it) and is served from the UNFILTERED (union) tier (the filtered tier is a lead-study subset, not a union).
3. **Service layer**: `app/services/` contains data access classes (inheriting from `GCloudTabixBase`) that handle tabix queries, parsing, and result formatting. Metadata harmonization uses `build_harmonizer_config(dataset_id)` from the registry to get harmonizer settings. `GeneNameAndPositionMapping` provides gene coordinate lookups (per GENCODE version) used by `SearchIndex` and chromatin peaks streaming to enrich results with genomic positions. `GeneGroupService` (`app/services/gene_group_service.py`) reads the three HGNC gene-group CSVs with polars' `read_csv` (all columns as strings via `infer_schema_length=0`, then ids coerced explicitly) at startup and builds an in-memory map from `hgnc_id` to its FULL gene-group lineage (leaf family plus all ancestor families via the precomputed transitive closure), plus the inverse `group_id -> member hgnc_ids` for descendant resolution. The three inputs are HGNC-native comma-separated CSVs: `gene_has_family` (`hgnc_id`, `family_id` leaf links), `hierarchy_closure` (already-transitive `child_fam_id`/`parent_fam_id`/`distance` rows; only proper ancestors with `distance > 0` are kept, the leaf is unioned in from `gene_has_family`), and `family` (`id` -> `name`, also indexed lowercase for case-insensitive name lookups). File paths come from the active profile's `genes` config keys `gene_has_family_file`, `hierarchy_closure_file`, `family_file` (alongside the existing `hgnc_file`). These three keys are present in BOTH profiles — `app/config/profiles/finngen/genes.py` (under `gs://finngen-commons/results_api_data/mapping_files/`) and `app/config/profiles/daly/genes.py` (under `gs://daly-genetics-results/mapping_files/`) — pointing at `hgnc_gene_has_family.csv`, `hgnc_hierarchy_closure.csv`, and `hgnc_family.csv`, which must be uploaded to each profile's GCS `mapping_files/` directory for the feature to activate. Loading is resilient: if the paths are unconfigured, or a file is missing or unreadable (e.g. not yet uploaded to GCS), a warning is logged and the group map is left empty so startup never fails (mirroring the coordinate-loading try/except in `SearchIndex._load_genes`). Accessors: `group_ids_for_hgnc_id(hgnc_id)`, `groups_for_hgnc_id(hgnc_id)` -> `[(group_id, group_name)]`, `members_of_group(group_id=..., group_name=...)` -> set of member `hgnc_id`s (any leaf/ancestor/root group resolves to all descendant members), `resolve_group_id(group_name)`, `group_name(group_id)`, and `is_loaded()`. Wired through the service container as `gene_group_service` (`create_gene_group_service` in `app/core/service_container.py`) and exposed via `get_gene_group_service()` in `app/dependencies.py`. To turn the `hgnc_id`s returned by `members_of_group` into symbol/ensembl/coordinates, `SearchIndex._load_genes` now also stores each gene record's `hgnc_id` and builds a `genes_by_hgnc_id` lookup (accessor `get_gene_by_hgnc_id(hgnc_id)`); this is the single bridge from HGNC ids back to the symbol/ensembl/coords already loaded from the same HGNC complete-set + GENCODE source (no coordinate logic is duplicated, and chrom keeps the API-wide X=23 convention since it comes from the GENCODE coords lookup). **HGNC id format**: the gene-group files use BARE numeric ids (`3023`) while the complete set (and thus `genes_by_hgnc_id`) uses the prefixed `HGNC:3023` form, so `GeneGroupService` canonicalizes every id to the `HGNC:NNNN` form (via `_canonical_hgnc_id`) when building its maps, and `get_gene_by_hgnc_id` also tolerates the bare form on lookup. Without this, every member resolved to null symbol/coords. The `gene-groups` router (`app/routers/gene_groups.py`, registered in `app/server.py`) joins these two services: it requires exactly one of `group_id`/`group_name` (400 otherwise), resolves a name case-insensitively (404 if unknown, both for an unresolvable name and for an unknown id), and returns `{group_id, group_name, exclude_olfactory, count, members:[{hgnc_id, symbol, ensembl_id, chr, gene_start, gene_end}]}` sorted by symbol (null symbols last). Members lacking a search/coords record are still listed with null fields rather than dropped, and if `is_loaded()` is False (group files not yet uploaded) it logs and returns an empty member list (count 0) instead of erroring. Olfactory receptors are GPCRs that dominate large families (e.g. the G protein-coupled receptor group) by sheer count; the optional `exclude_olfactory` query param (default False on the raw endpoint; the mcp-server tool defaults it to True) drops any member whose lineage contains the `Olfactory receptors` group via `members_of_group(..., exclude_olfactory=True)`. The same router also exposes `GET /gene/normalize?symbols=<comma-separated>`, which resolves each input gene symbol/alias/previous symbol to its current approved HGNC symbol and returns `{mappings:[{input, approved, matched_on}], unresolved:[...]}`. It reuses `SearchIndex.normalize_symbol`, which performs an EXACT case-insensitive dict lookup (built once, lazily, from `self.genes`' approved `symbol` plus merged `aliases`/previous symbols — the free-text gene `name` and ensembl id are deliberately excluded) rather than the fuzzy `search()` path, so normalization never yields ranked near-miss false positives. `matched_on` is `approved` when the input is already an approved symbol, else `alias_or_previous` (the HGNC load merges alias and previous symbols without retaining which list a match came from).
4. **Routing layer**: `app/routers/` defines endpoints that validate input, call services, and return formatted responses

### Deployment

- Docker container built with htslib (for tabix) and gcloud SDK
- GCS authentication via Workload Identity or service account key. `GCloudTabixBase.ensure_gcs_token()` fetches/refreshes the OAuth token used by the remaining tabix subprocesses (`tabix -H` headers, gnomad) — exported as `GCS_OAUTH_TOKEN`; the whole check-and-refresh (including first init) holds `_gcs_token_lock` so concurrent first calls can't race on the env var. Range reads instead authenticate via the pooled aiohttp/`gcloud-aio` session (`storage._headers()`). htslib reports transient GCS access failures (timed-out/partial HTTPS requests) as misleading `Invalid argument` / `No such file or directory` / index-load errors and never retries them, so `_get_header` retries `tabix -H` up to `_TABIX_MAX_ATTEMPTS` (3) times with exponential backoff; the in-process range fetchers (`_fetch_full` for the `.tbi`, `_fetch_blocks` for data) retry the same way on transient errors. The aiohttp GCS client + `gcloud-aio` `Storage` are created lazily via `_ensure_storage()` on first use (now also used by the range path) and recreated if a prior session was closed. Header retrieval has both a blocking (`_get_header`, `subprocess.run`) and a non-blocking async (`_get_header_async`/`_cache_header_async`, `asyncio.create_subprocess_exec`) form; the data-access objects no longer fetch headers eagerly in `__init__` (which blocked the event loop on a `tabix -H` per file) — instead the header is fetched lazily by `get_header()` and prefetched (non-blocking) at startup by each object's `warm()` (`tabix -H` also downloads the `.tbi` index, warming that cache too).
- **In-process tabix range reads** (`app/services/tabix_query.py` + `GCloudTabixBase._stream_range`): the dominant cost of large variant batches was htslib issuing one HTTP byte-range request per region **serially** over a single connection (~20 ms/region in-region — the same query on a local file copy is ~36× faster, and the API VM is co-located with the bucket, so it is round-trip latency, not bandwidth or WAN). The replacement: (1) the bgzf-compressed `.tbi` index is fetched once via the pooled session and parsed into bin/linear/chunk structures (cached in-process per file; the raw `.tbi` is also persisted to the existing `/tmp/tbi_cache`); (2) query regions map (via `reg2bins` + linear-index min-offset) to compressed chunk byte ranges, merged so each boundary block is read once; (3) `_fetch_blocks` streams each range over its instance's aiohttp session with a `Range:` header and stops after the final needed bgzf block (avoiding a fixed 64 KiB over-read per range); (4) decompress (`zlib`, gzip/bgzf member-by-member) + record filtering run in a **`ProcessPoolExecutor`** (`_get_filter_pool`, `min(4, cpu-1)` workers, override `TABIX_FILTER_WORKERS`) split into contiguous sub-batches, because the per-line parsing is GIL-bound and would otherwise pin one core while the others idle — this is what makes the all-resources batch parallelize.
  - **Bounding open sockets / file descriptors**: each data-access object owns its own aiohttp session+connector (gcloud-aio's `Storage`/`Token` binds a token session per instance, so a *shared* connector breaks token refresh with "Event loop is closed"). To stop a fan-out from opening thousands of sockets at once — which exhausted the process FD limit ("Too many open files"), since an all-resources request touches ~12-15 files — every range fetch acquires a **process-wide `asyncio.Semaphore`** (`_get_fetch_semaphore()`, size `_GCS_MAX_CONNECTIONS`=128, override `GCS_MAX_CONNECTIONS`). This caps *total* concurrent fetches (hence sockets) across all files **and overlapping requests**, so socket count stays bounded regardless of request concurrency (measured peak ~370 for two concurrent 888-variant all-resources requests vs ~730 for a single request before the cap). Connector `keepalive_timeout` is 5 s so idle sockets from a burst free quickly. This is purely a per-request concurrency spike, not an FD leak — idle FD count returns to baseline. The container entrypoint (`start.sh`) also raises `ulimit -n` to 65536 as defense-in-depth, and the k8s deployment sets `TABIX_FILTER_WORKERS`/`GCS_MAX_CONNECTIONS`. The filter has a fast path for point queries on point records (byte-level `(seqname, pos)` set membership, no int parsing) and a general interval-overlap path; output is the matching record lines in coordinate order, identical to `tabix -R` (verified against `tabix` and the integration suite). Net measured: 888-variant all-resources ~30 s → ~12 s, single-resource ~18 s → ~4–5 s, small batches ~1.5 s → ~0.3 s. Services that use `_stream_range` must let `GCloudTabixBase._init_storage` create the session (do not override it to a no-op as the gnomad path, which uses only its own tabix subprocess, still does).
- Default port: 4000
- Entry point: `/opt/genetics-results-api/start.sh` which initializes gcloud auth and starts uvicorn
- **Single-process startup (no reload by default)**: `run_server.py` runs uvicorn with `reload=False` unless `RELOAD` is truthy (`1`/`true`/`yes`). With reload on (local dev convenience), uvicorn serves from a separate worker subprocess that has its OWN service container, so all warming done in `run_server.py`'s `__main__` validations is discarded and re-done lazily in the worker — slow startup and a slow first request per service. With reload off (production default), the serving process IS the process that ran `__main__`, so the `__main__` validations' warmed singletons (data access objects + headers, gene name/position maps, gene-disease) are reused directly and the `lifespan` warming is mostly a fast no-op. Local dev opts into reload with `RELOAD=1`.
- **Startup data-file verification**: before uvicorn starts, `run_server.py` calls `app/services/startup_checks.py::verify_all_data_files()`, which fails startup if any configured data file is missing/unreadable. It runs concurrently (thread pool) and collects ALL failures before raising one error: (1) `tabix -H` on every fixed/combined tabix file enumerated from config — cs `all_cs_file`/`all_cs_qtl_file`, exome `all_exome_file`, gene-based `file`, coloc `credset_file`/`coloc_file`, fixed sumstats `file`, expression, chromatin peaks, variant-annotation, gnomad, rsid (paths shared across datasets are deduped; this also proves the tabix index loads), and (2) fsspec existence checks for non-tabix mapping files (genes `model_file`/`gene_name_mapping_file`/`hgnc_file`/per-version gencode position files, gene-disease files). Per-phenotype `prefix`+`suffix` files are not enumerable, so they are not checked here (this includes IBD exome, which has only per-phenotype files and no combined file). Intentionally excluded: the gene-group CSVs (loaded resiliently by design), `variant_set_files` (configured but consumed by no service), and the per-request `phenotype_markdown_template`.
- **Startup cache warming**: the FastAPI `lifespan` (which runs inside the uvicorn serving process) warms two groups concurrently via a single `asyncio.gather`, so total startup time is the slower of the two rather than their sum: (1) a worker thread (`asyncio.to_thread`) preloads the sync, GCS-backed singletons in dependency order — `gene_disease_data`, then `gene_name_mapping` (the GENCODE/HGNC gene name→coordinate maps that every `*_by_gene` endpoint needs), then `search_index`; running them off the event loop keeps the blocking GCS reads from stalling the loop; (2) `warm_all()` on the credible-set (`DataAccess`), coloc (`DataAccessColoc`), expression (`DataAccessExpression`), and chromatin-peaks (`DataAccessChromatinPeaks`) services, which construct every gcloud tabix data-access object and call its `warm()` to prefetch headers and the `.tbi` index cache (non-blocking). Together these mean no request — variant-, region-, OR gene-keyed — pays a blocking GCS read on first use. (Sumstats serves per-phenotype files that aren't enumerable, so it isn't warmed — same reason `verify_all_data_files()` skips them.) Per-tabix-file failures are logged and skipped (`verify_all_data_files()` remains the authoritative reachability gate); a failure loading a sync singleton aborts startup loudly, as intended. With reload off (production), the `__main__` validations have usually already constructed these singletons in-process, so this `lifespan` warming reuses them and only `search_index` is loaded fresh.
- **Parallel fan-out**: every cross-resource range merge (credible sets, coloc, expression, chromatin peaks, and per-phenotype sumstats) primes each file's first tabix read concurrently via `start_iterators()` (`app/core/streams.py`) before handing the iterators to `merge()`; the credible-set `DataAccess.stream_range` additionally opens its per-file access objects/streams concurrently with `asyncio.gather`. This is necessary because asyncstdlib's `merge()` seeds its heap by awaiting the first element of each iterable one-by-one, which would otherwise serialize the network-bound per-file first reads (the dominant fan-out latency). Empty iterators are dropped as before, and an iterator whose first read fails re-raises lazily when consumed, preserving the original error semantics.
